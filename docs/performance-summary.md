## Performance data (local, 100 concurrent threads)

Source data: `docs/perf-results.json` generated by `scripts/perf/run_perf.js`.

### Scenarios (professor-required)

- **Scenario A**: job search + job detail view
- **Scenario B**: apply submit (DB write + Kafka produce)

### Results (summary)

- **Scenario A (cache cold)**: avg \(368.3\) ms, p95 \(1777.1\) ms
- **Scenario A (cache warm)**: avg \(308.3\) ms, p95 \(472.6\) ms
- **Scenario B (apply submit)**: avg \(403.4\) ms, p95 \(1018.4\) ms

### Justification of results

- **Caching impact (Scenario A)**:
  - Warm-cache runs reduce p95 significantly because repeated searches/details hit Redis-backed cached lookups instead of repeated MySQL queries.
  - p50 can shift based on connection pooling, host scheduling, and “first hit” cache fill overhead; p95 is the key stability signal under concurrency.

- **Kafka impact (Scenario B)**:
  - Application submit includes a synchronous DB write plus a Kafka produce step; under load this increases tail latency (p95/p99).
  - The key requirement is correctness under at-least-once semantics (idempotency) rather than the lowest p95 latency.
